# Job Queue Engine — BullMQ From Scratch

> **Goal:** Build a production-grade job queue engine from scratch using Node.js + TypeScript + Redis — with worker concurrency, stall detection, Lua atomic scripts, rate limiting, and a real-time [Socket.IO](http://Socket.IO) dashboard. Zero shortcuts. Every tradeoff understood from first principles.
>

---

# 🧠 What This Engine Does

A job queue engine accepts jobs from producers, stores them durably in Redis, and distributes them to workers reliably — even when workers crash, Redis restarts, or the system is hammered with 10,000 jobs at once. This is the internals of what BullMQ, Sidekiq, and Celery are built on.

Companies like Shopify, GitHub, and Stripe run millions of background jobs daily. Understanding this at the implementation level is a rare skill that almost no fresher has.

---

# 🏗️ Architecture Overview

```
Producer (your app)
     ↓ queue.add('send-email', data)
[Redis Sorted Set — waiting]
     ↓ Scheduler polls every 500ms
[Redis Hash — active] ← Worker locks job here
     ↓ Worker processes
[Redis Hash — completed] or [Redis Sorted Set — failed/DLQ]
     ↑
Watchdog scans active every 10s → stale heartbeat → move back to waiting
     ↑
Dashboard subscribes to Redis Pub/Sub → Socket.IO → Real-time UI
```

# ⚙️ Tech Stack

| Layer | Technology | Why |
| --- | --- | --- |
| Runtime | Node.js v20 + TypeScript | Your stack, strong typing for complex state |
| Redis Client | ioredis | Lua script support via defineCommand() |
| Atomic Ops | Lua scripts | Multi-step Redis ops without race conditions |
| Concurrency | Semaphore pattern | Gate how many jobs run simultaneously |
| Real-time | [Socket.IO](http://Socket.IO) | Push job events to dashboard |
| Dashboard | HTML + Vanilla JS | No framework needed for simple UI |
| Testing | Jest + Testcontainers | Real Redis in tests, not mocks |
| Infra | Docker Compose | Redis + 3 worker replicas |

# 🎤 Interview Talking Points

**"Why did you build a job queue from scratch?"**

Because using BullMQ without understanding it means you can't debug it in production. I needed to understand why Lua scripts are used, what stall detection solves, and how concurrency works. Building it from scratch means I can explain every design decision from first principles.

**"How do you prevent duplicate job processing?"**

The moveToActive Lua script atomically removes a job from the waiting Sorted Set and adds it to the active Hash in a single Redis operation. Because Redis is single-threaded, no two workers can execute this script simultaneously for the same job — one will get the job, the other gets nil.

**"What happens when a worker crashes mid-job?"**

The Watchdog process scans the active Hash every 10 seconds looking for jobs where heartbeatAt is stale (>30s old). When it finds one, it atomically moves the job back to the waiting Sorted Set with an incremented attempt count. The job will be retried by the next available worker.

**"How does rate limiting work?"**

I implemented a token bucket algorithm in a Lua script. Each queue has a token count and a lastRefill timestamp in Redis. On each job pickup, the script checks if enough time has passed to refill tokens, decrements one token, and returns whether the job can proceed. The whole check-and-decrement is atomic.
